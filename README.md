## Sprint Planning

### Sprint 1: Minimum Viable Product

Sprint 1 focuses on developing the basic working version of the task tracking system.

#### Sprint 1 Features
- User registration
- User login
- Create task
- View task list

#### Sprint 1 Goal
To deliver a minimum working version where users can register, log in, create tasks, and view their task list.

---

### Sprint 2: Feature Enhancement

Sprint 2 focuses on improving and extending the features developed in Sprint 1.

#### Sprint 2 Features
- Edit task
- Delete task
- Mark task as completed
- Filter tasks by status
- Improve user interface
- Improve validation

#### Sprint 2 Goal
To enhance the application by allowing users to manage tasks more effectively with better functionality and usability.

---

## Iterative and Incremental Development

This project applies iterative and incremental development by delivering the system in two stages.

In **Sprint 1**, the team develops the minimum viable product with basic features such as registration, login, task creation, and task viewing.

In **Sprint 2**, the team improves the system by adding task editing, deletion, completion status, filtering, validation, and user interface improvements.

This approach allows the team to build the application step by step while continuously improving the system after each sprint.

## GitHub Collaboration Strategy

The team uses GitHub to manage version control and collaboration.

### Branching Strategy

The `main` branch is used as the stable branch. Each feature is developed in a separate feature branch before being merged into `main`.

Example branches:

- `feature-register`
- `feature-login`
- `feature-create-task`
- `feature-view-task`
- `feature-edit-task`
- `feature-delete-task`
- `feature-task-status`
- `feature-filter-task`

---

## GitHub Issues

Each team member is assigned at least two GitHub issues.

Example issues:

- Setup GitHub repository and README
- Design login and register page
- Develop create task feature
- Develop task list display
- Develop edit task feature
- Develop delete task feature
- Implement task completion status
- Implement task filtering by status

## Pull Request Workflow

The team follows this pull request workflow:

1. Create a feature branch.
2. Develop the assigned feature.
3. Commit changes regularly.
4. Push the branch to GitHub.
5. Create a pull request.
6. Another team member reviews the pull request.
7. After approval, merge into the `main` branch.

This workflow helps the team maintain code quality and show evidence of collaboration.
